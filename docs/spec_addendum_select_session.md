# High & Low 実装仕様書 補足：select のセッション取得

本ドキュメントは、メイン仕様書（PDF）の **7.3**（Request Body に `session_id` 必須）と **7.3.1**（「user_id をキーとしてゲームセッションを取得する」）の解釈を統一する補足です。

---

## 1. 不整合の内容

| 箇所 | 内容 |
|------|------|
| 7.3 Request Body | `session_id`（必須）・`ver`（必須）をクライアントが送信する。 |
| 7.3.1 処理手順 | 「**user_id をキーとして**ゲームセッションを取得する」と記載。 |

このままでは「取得キーは user_id か session_id か」が曖昧になるため、以下で一意に定める。

---

## 2. 確定仕様（取得キーと session_id の役割）

### 2.1 セッション取得

- **取得キー**: **user_id**（認証済みユーザー ID）とする。
- サーバーは **user_id をキーに** 1 件のゲームセッションを取得する（1 ユーザー 1 セッションの前提）。
- 実装例: `FindByUserID(userID)` でセッションを取得する。

### 2.2 Request Body の session_id の役割

- クライアントは **7.3 どおり** Request Body に `session_id`（必須）と `ver`（必須）を含める。
- **session_id の用途**: 取得したセッションの ID（`game.ID`）と、クライアントが送った `session_id` が **一致するか** の検証に用いる。
- 一致しない場合（他セッションを指定したなど）は **403 forbidden** を返す。
- これにより「user_id で取得」と「Body に session_id 必須」の両立と、クライアント・サーバー間の対象セッションの一致を保証する。

### 2.3 処理フロー要約

1. 認証から **user_id** を取得する。
2. **user_id** をキーにゲームセッションを 1 件取得する。
3. セッションが存在しない → **404**（セッション不存在）。
4. 取得した `game.ID` と Request Body の **session_id** を比較する。
5. 一致しない → **403 forbidden**。
6. 一致する場合、以降は 7.3 の手順（remaining 計算・カード決定・勝敗判定など）に従う。

---

## 3. 仕様書との対応

- **7.3.1**「user_id をキーとしてゲームセッションを取得する」→ 上記 2.1 のとおり **user_id** を取得キーとする。
- **7.3** Request Body の **session_id（必須）**→ 上記 2.2 のとおり **一致検証** に使用する。
- これにより 7.3 と 7.3.1 の記載は矛盾せず、実装も一意に定まる。
